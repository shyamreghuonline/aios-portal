# AIOS Portal — Password-Based Authentication Flow

## Overview

AIOS Portal uses **Firebase Authentication with email-password accounts** for both **Admin** and **Student** users. Students log in using their **Enrollment ID** (e.g., `AIOS26AP000001`) as the username, paired with a password they set via a secure SMS link.

---

## 1. User Types & Credentials

| Role   | Username format                          | Password source                               |
|--------|------------------------------------------|-----------------------------------------------|
| Admin  | `aiosadmin` (or any admin email)        | Manually configured in `auth-context.tsx`     |
| Student| `studentId` (e.g., `AIOS26AP000001`)    | Set by student via admin-issued SMS link      |

Internally, every student account is stored in **Firebase Auth** as an email-password user with a synthetic email:

```
studentId@aiosedu.local
```

This mapping lets us use Firebase's robust email-password infrastructure while presenting a student-ID-based login to users.

---

## 2. Admin Login Flow

1. Admin opens the portal login page (`/`).
2. Selects **Admin** tab.
3. Enters username (`aiosadmin`) and password.
4. Frontend calls `signInWithEmailAndPassword(auth, email, password)`.
5. `auth-context.tsx` detects the Firebase user's email and checks it against `ADMIN_EMAILS`.
6. If matched, the user is granted `role: "admin"` and redirected to `/admin`.

---

## 3. Student Login Flow

1. Student opens the portal login page (`/`).
2. Selects **Student** tab (default).
3. Enters **Enrollment ID** and **Password**.
4. Frontend builds the synthetic email:
   ```ts
   const email = `${studentId}@aiosedu.local`;
   ```
5. Calls `signInWithEmailAndPassword(auth, email, password)`.
6. `auth-context.tsx` detects `@aiosedu.local` suffix and:
   - Extracts the `studentId`.
   - Queries Firestore `students` collection where `studentId == <id>`.
   - Attaches the matching student record to the auth user object.
7. Student is redirected to `/student`.

**Security notes:**
- Firebase Auth handles brute-force protection (rate limiting, account lockout).
- No phone-OTP or reCAPTCHA is required, eliminating Firebase Phone Auth billing/credential issues.

---

## 4. New Student Onboarding (Password Creation)

When an admin creates a student in `/admin/students/new`:

1. **Student record** is saved to Firestore (`students` collection, keyed by phone number).
2. **Password token** is generated by calling `POST /api/auth/create-password-token`:
   - Creates a cryptographically secure random token.
   - Stores it in Firestore `passwordTokens` collection with:
     - `studentId`
     - `phone`
     - `createdAt` (server timestamp)
     - `expiresAt` (+24 hours)
     - `used: false`
   - Returns a one-time setup link:
     ```
     https://<APP_URL>/set-password?token=<TOKEN>
     ```
3. **SMS** is sent via `POST /api/send-sms` with `type: "password-link"`:
   - Includes the student's name, Enrollment ID, and the setup link.
   - Link is valid for **24 hours**.

---

## 5. Student Sets Password (`/set-password`)

1. Student clicks the link in the SMS.
2. `/set-password/page.tsx` reads `?token=` from the URL.
3. On mount, it validates the token via `POST /api/auth/validate-token`:
   - Checks existence, expiry, and `used === false`.
   - Returns the associated `studentId` and `phone`.
4. Student enters and confirms a new password.
5. On submit, calls `POST /api/auth/set-password`:
   - Validates the token again server-side.
   - Creates or updates the Firebase Auth user with email `studentId@aiosedu.local`.
   - Sets the password using Firebase Admin SDK (`updatePassword` or `createUser`).
   - Marks the token as `used: true` in Firestore.
6. On success, the page automatically signs the student in via `signInWithEmailAndPassword` and redirects to `/student`.

---

## 6. Admin-Mediated Password Reset

Admins can reset a student's password from `/admin/students`:

1. Admin clicks the **Reset Password** button in the students table or student detail modal.
2. Frontend calls `POST /api/auth/create-password-token` for that student.
3. Then calls `POST /api/send-sms` (`type: "password-link"`) with the new link.
4. Student receives a fresh SMS with a new 24-hour link to set their password.

**Why admin-mediated?**
- Prevents students from initiating unlimited reset loops (brute-force / social-engineering risk).
- Ensures the admin verifies identity before re-issuing access.
- Firebase's built-in rate limiting still applies at the auth layer.

---

## 7. API Routes

| Route                              | Method | Purpose                                         |
|------------------------------------|--------|-------------------------------------------------|
| `/api/auth/create-password-token`  | POST   | Generates a secure one-time token & setup link  |
| `/api/auth/validate-token`          | POST   | Validates token (existence, expiry, unused)       |
| `/api/auth/set-password`            | POST   | Creates/updates Firebase Auth password & burns token |
| `/api/send-sms`                   | POST   | Sends SMS; supports `type: "password-link"`       |

---

## 8. Firestore Collections

### `students`
- Document ID: phone number (e.g., `+919876543210`)
- Fields: `studentId`, `name`, `phone`, `email`, `course`, `university`, `totalFee`, `createdAt`, etc.

### `passwordTokens`
- Document ID: random token string
- Fields:
  - `studentId: string`
  - `phone: string`
  - `createdAt: Timestamp`
  - `expiresAt: Timestamp`
  - `used: boolean`

## 9. Environment Variables

Ensure these are set in `.env.local`:

```bash
# Firebase Admin SDK
FIREBASE_ADMIN_PROJECT_ID=...
FIREBASE_ADMIN_CLIENT_EMAIL=...
FIREBASE_ADMIN_PRIVATE_KEY=...

# Firebase Client
NEXT_PUBLIC_FIREBASE_API_KEY=...
NEXT_PUBLIC_FIREBASE_AUTH_DOMAIN=...
NEXT_PUBLIC_FIREBASE_PROJECT_ID=...

# App URL (used for password setup links)
NEXT_PUBLIC_APP_URL=https://your-portal-domain.com

# SMS Provider: MSG91 (India-focused, cheaper)
SMS_PROVIDER=msg91
MSG91_AUTH_KEY=your_msg91_auth_key
MSG91_SENDER_ID=AIOSDB
```

## 10. Security Checklist

- [ ] Firebase Auth email enumeration is mitigated by generic error messages on the login page.
- [ ] Tokens are single-use and expire after 24 hours.
- [ ] Tokens are generated server-side using `crypto.randomBytes`.
- [ ] All token validation happens server-side before any password change.
- [ ] Admin-only password reset prevents unauthenticated reset loops.
- [ ] Firebase handles password hashing, rate limiting, and transport security (HTTPS).

---

## 11. Troubleshooting

| Problem                                  | Likely Cause                                    | Fix                                        |
|------------------------------------------|-------------------------------------------------|--------------------------------------------|
| "Invalid Student ID or password"        | Wrong credentials or student not yet set password| Admin should send a new password link SMS  |
| "Token expired or already used"         | Student waited >24h or reloaded the page        | Admin should send a new password link SMS  |
| SMS not received                        | SMS provider misconfigured or quota exhausted   | Check `.env.local` SMS credentials         |
| "User not found" during set-password    | Token valid but Auth user missing (rare)      | `/api/auth/set-password` auto-creates user   |
